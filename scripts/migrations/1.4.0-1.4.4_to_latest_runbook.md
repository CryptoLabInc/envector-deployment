# 1.4.0–1.4.4 → latest In-Place Upgrade — Deployment Runbook

- **마이그레이션 파일**: `1.4.0-1.4.4_to_latest.sql`
- **자매 스크립트**: `1.4.5-1.4.6_to_latest.sql` (1.4.5–1.4.6 전용)
- **대상 범위**: stable 태그 `1.4.0` ~ `1.4.4`. `1.4.5`–`1.4.6`은 자매 스크립트로 처리.

---

## 0. 적용 가능 여부 판정

이 스크립트는 **1.4.0–1.4.4 범위** 전용입니다. 적용 전 현재 배포 버전을 확인하십시오.

| 현재 버전 | 경로 |
|----------|------|
| `v1.0.x` | **불가** — 구조 비호환(`_Items`/`_ev`). export/re-ingest 필요. |
| `v1.1.0`–`v1.2.2` | **불가** — blob 포맷 비호환(ES2 monolith). export/re-ingest 필요. |
| `1.4.0`–`1.4.4` | ✅ **이 스크립트** |
| `1.4.5`–`1.4.6` | `1.4.5-1.4.6_to_latest.sql` 사용 |

## 1. 배경

1.4.0-1.4.4 배포를 HEAD 스키마로 끌어올리는 두 가지 작업을 한 윈도우에서 수행합니다.

1. **글로벌 델타 (PART A)** — 1.4.5에서 처음 등장한 객체를 보강:
   - `indexes.ct_trunc`, `indexes.import_id`
   - `shards.owner_merge_task_id`
   - `keys.status`
   - `import_log` 테이블 (+ 유니크 인덱스 + CHECK)
   - named 인덱스 3종 (`idx_indexes_import_id`, `idx_keys_status_transient`, `idx_task_queue_target_shard_id`)
2. **per-index 변환 (PART B)** — `<idx>_shard_map`(item_id PK + id_in_shard)을
   `<idx>_item_shard_map`(복합 PK `(shard_id, item_id)` + row_idx)으로 전환(#1888 리팩토링)
   + DeleteData partial 인덱스 + `<idx>_item_merge_lock` 백필.

⚠️ **이 SQL은 신 application binary와 한 배포 단위로 묶여야 합니다.** 구 binary는 변환 후
스키마(rename + row_idx)를 읽지 못하고, 신 binary는 변환 전 스키마에서 첫 런타임 쿼리가
실패합니다. **간소화 이후 binary에는 부팅 시 self-heal 로직이 없으므로**, 이 스크립트가
유일한 업그레이드 경로입니다.

## 2. 사전 요구사항

- [ ] 현재 배포가 **1.4.0–1.4.4** 임을 확인 (§0).
- [ ] 신 application binary 빌드/배포 준비 완료 (HEAD 스키마 기대).
- [ ] 마이그레이션 윈도우 확보. PART B는 인덱스당 수 분 ~ 수십 분.
  - Step 1b(per-row assertion gate)와 Step 3(ROW_NUMBER 백필)은 각각 `<idx>_shard_map`
    전체를 한 번씩 스캔하므로, 대형 인덱스(> 10M rows)는 추가 시간 고려.
- [ ] DB backup 확보 (24h 롤백 윈도우 밖 재해 시 유일한 복구 경로).

⚠️ **구 binary와 변환 후 schema는 호환 불가.** binary와 schema가 짝을 이뤄야 합니다.

## 3. 진행 순서

### 단계 1: 트래픽 일시 정지

```bash
docker compose stop envector-backend
```

> 전 구간 app-stopped 전제이므로 read-only freeze가 아닌 완전 정지를 권장합니다.
> (PART B의 RENAME이 구 binary를 즉시 깨뜨립니다.)

### 단계 2: 글로벌 델타 적용 (PART A — 1회)

PART A는 전부 `IF NOT EXISTS`라 멱등이며, 한 번만 실행하면 됩니다.

```bash
# PART A 블록만 추출하여 1회 실행 (또는 단계 3의 루프가 멱등으로 재실행 — 무해)
psql "$DATABASE_URL" -v ON_ERROR_STOP=1 -f 1.4.0-1.4.4_to_latest.sql
# ↑ PART B 템플릿의 <index_name> 미치환분은 첫 인덱스명이 없으면 에러가 나므로,
#   실제로는 단계 3의 sed 루프로 PART A+B를 함께 돌리는 방식을 권장(아래).
```

> PART A를 단독 실행하려면 파일에서 `PART A` 섹션(첫 `BEGIN;`~첫 `COMMIT;`)만 잘라
> 실행하십시오. 단계 3의 루프를 쓰면 PART A가 인덱스마다 재실행되지만 멱등이므로
> 첫 인덱스 이후 no-op입니다.

### 단계 3: per-index 변환 (PART B — 인덱스마다)

인덱스 목록 확인:

```sql
SELECT lower(index_name) AS index_name FROM indexes ORDER BY index_name;
```

자동화 (PART A가 멱등이므로 전체 파일을 인덱스마다 파이프해도 안전):

```bash
psql "$DATABASE_URL" -tAc "SELECT lower(index_name) FROM indexes" \
  | while read idx; do
      echo ">>> migrating index: $idx"
      sed "s/<index_name>/$idx/g" 1.4.0-1.4.4_to_latest.sql \
        | psql "$DATABASE_URL" -v ON_ERROR_STOP=1
      psql "$DATABASE_URL" -c "ANALYZE ${idx}_item_shard_map;"
    done
```

> 각 인덱스 블록은 `BEGIN ... COMMIT`으로 원자적입니다. 한 인덱스가 실패하면 해당
> 인덱스만 롤백되고 나머지는 영향 없습니다(§5.3). `ON_ERROR_STOP=1`로 실패를 즉시 감지.

> **Step 1b가 발화하면**(`id_in_shard <> item_id-sorted rank`): 저장된 slot 번호가
> item_id 오름차순 rank와 어긋난다는 뜻입니다. Step 3 백필이 같은 rank로 row_idx를
> 재계산하므로, 어긋난 채 진행하면 row_idx가 실제 blob slot과 불일치해 검색이 깨질 수
> 있습니다. **발화 = 즉시 export/re-ingest 확정은 아닙니다.** 먼저 어긋난 행이 어느
> shard인지(아래 쿼리), 그리고 그것이 오래 deprecated된 행에 국한되는지 확인하십시오.
> 라이브 행까지 어긋나면 그 인덱스는 v1.1.0-v1.2.2로 간주해 export/re-ingest 하고, 나머지
> 인덱스는 계속 진행합니다.
>
> ```sql
> -- 윈도우 함수는 WHERE에 직접 못 쓰므로 서브쿼리로 계산 후 필터.
> SELECT shard_id, item_id, id_in_shard, computed, deprecated
> FROM (
>     SELECT shard_id, item_id, id_in_shard, deprecated,
>            ROW_NUMBER() OVER (PARTITION BY shard_id ORDER BY item_id) - 1 AS computed
>     FROM <index_name>_shard_map
> ) t
> WHERE id_in_shard <> computed
> ORDER BY shard_id, item_id;
> ```

### 단계 4: 검증 (각 인덱스마다 + 글로벌)

```sql
-- (a) 신 테이블 스키마
\d <index_name>_item_shard_map
-- 기대값: PK (shard_id, item_id); columns {shard_id, item_id, row_idx, deprecated, created_at, updated_at}
--         id_in_shard 컬럼 없음

-- (b) _old 보존 확인 (24h 롤백 윈도우)
SELECT EXISTS (SELECT 1 FROM information_schema.tables WHERE table_name = '<index_name>_shard_map_old');
-- expect: true

-- (c) legacy 부재
SELECT EXISTS (SELECT 1 FROM information_schema.tables WHERE table_name = '<index_name>_shard_map_legacy');
-- expect: false (1.4.0-1.4.4는 애초에 없음)

-- (d) merge-lock 백필 확인
SELECT EXISTS (SELECT 1 FROM information_schema.tables WHERE table_name = '<index_name>_item_merge_lock');
-- expect: true

-- (e) DeleteData partial 인덱스 확인 (row_idx_alive는 의도적으로 미생성)
SELECT indexname FROM pg_indexes
WHERE tablename = '<index_name>_item_shard_map'
ORDER BY indexname;
-- expect 포함: idx_<index_name>_item_shard_map_item_id / _alive / _deprecated
-- expect 불포함: idx_<index_name>_item_shard_map_row_idx_alive
SELECT indexname FROM pg_indexes WHERE tablename = '<index_name>_ctxt_map' AND indexname = 'idx_<index_name>_ctxt_map_deprecated';
-- expect: 1 row

-- (f) row_count invariant
SELECT
    (SELECT COUNT(DISTINCT item_id) FROM <index_name>_item_shard_map WHERE deprecated = FALSE) AS distinct_alive,
    (SELECT row_count FROM indexes WHERE index_name = '<index_name>') AS index_row_count;
-- expect: distinct_alive = index_row_count
```

글로벌 (1회):

```sql
SELECT column_name FROM information_schema.columns
WHERE (table_name = 'indexes' AND column_name IN ('ct_trunc', 'import_id'))
   OR (table_name = 'shards'  AND column_name = 'owner_merge_task_id')
   OR (table_name = 'keys'    AND column_name = 'status');
-- expect: 4 rows
SELECT to_regclass('import_log'); -- expect: import_log (non-null)
```

모든 인덱스가 (a)~(f) 통과해야 다음 단계.

### 단계 5: 신 binary 배포

> ⚠️ **반드시 단계 1~4(SQL)를 먼저 완료**한 뒤 신 binary를 배포하십시오.
> 간소화 이전 binary는 부팅 시 `ensureDeprecatedColumnsOnPerIndexTables`가
> `<idx>_item_shard_map`에 `ALTER TABLE ... ADD COLUMN`을 실행 — 테이블이 아직 옛 이름이면
> `relation does not exist`로 **부팅 실패**합니다. 간소화 이후 binary는 self-heal이 없으므로
> 변환 전 스키마에서 첫 런타임 쿼리가 실패합니다. 두 경우 모두 이 스크립트가 선행되어야 합니다.

```bash
docker compose up -d envector-backend
```

### 단계 6: 트래픽 재개 후 모니터링 (최소 24시간)

- ❌ "Item ID count mismatch" 에러 (있어선 안 됨)
- ❌ "relation does not exist" (변환 누락 인덱스)
- ✅ `indexes.row_count` == `COUNT(DISTINCT item_id)`
- ✅ 검색 latency 정상, 머지 cutover 정상

### 단계 7: 옛 테이블 정리 (24시간 이상 안정 운영 후)

```sql
DO $$
DECLARE idx TEXT;
BEGIN
    FOR idx IN SELECT lower(index_name) FROM indexes LOOP
        EXECUTE format('DROP TABLE IF EXISTS %I_shard_map_old', idx);
    END LOOP;
END $$;
```

```bash

```

### R2: 테이블 swap-back (각 인덱스마다)

```sql
BEGIN;
ALTER TABLE <index_name>_item_shard_map RENAME TO <index_name>_item_shard_map_failed;
ALTER TABLE <index_name>_shard_map_old  RENAME TO <index_name>_shard_map;
COMMIT;
```

### R3: 구 binary 재배포

```bash
docker compose up -d envector-backend:<previous-1.4.x-tag>
```

> PART A의 글로벌 ADD COLUMN/CREATE TABLE은 **additive**라 구 binary에 무해합니다.
> 롤백 시 되돌릴 필요 없습니다(구 binary는 신 컬럼을 단순히 안 읽음).

⚠️ 24시간 경과 후 `_old`가 삭제되면 DB backup 시점 복구만 가능.

## 5. Known Issues / 주의사항

### 5.1 대형 인덱스의 메모리/시간

Step 1b(assertion gate)와 Step 3(백필)는 `<idx>_shard_map` 전체를 ROW_NUMBER 윈도우로
스캔합니다. > 10M rows면 메모리/시간이 큽니다. 필요시 **shard_id 단위**로 청크 분할
(한 shard는 통째로 한 청크 — row_idx는 shard별 순번이므로 item_id로 자르면 깨짐):

```sql
INSERT INTO <index_name>_item_shard_map (shard_id, item_id, row_idx, deprecated, created_at, updated_at)
SELECT shard_id, item_id,
       ROW_NUMBER() OVER (PARTITION BY shard_id ORDER BY item_id) - 1,
       deprecated, created_at, updated_at
FROM <index_name>_shard_map
WHERE shard_id IN ( /* 이번 청크의 shard_id 목록 */ )
ON CONFLICT (shard_id, item_id) DO NOTHING;
```
청크 분할 시 Step 1b 게이트도 동일 shard_id 범위로 함께 검사하십시오.

### 5.2 통계 갱신

신 PK/인덱스 직후 planner가 빈 통계를 사용할 수 있으므로 단계 3 루프에 포함된
`ANALYZE <index_name>_item_shard_map;`를 반드시 수행하십시오.

### 5.3 다중 인덱스 부분 실패

각 인덱스가 독립 `BEGIN…COMMIT`이라 한 인덱스 실패가 다른 인덱스에 영향을 주지 않습니다.
단, **일부만 성공한 상태로 신 binary를 띄우면 실패한 인덱스의 검색이 깨집니다.** 실패한
인덱스만 골라 재시도하거나 일시 비활성화 후 후속 처리하십시오.

### 5.4 task_queue 컬럼은 손대지 않음

초기 directional 분석은 `task_queue`의 여러 컬럼을 1.4.0-1.4.4 델타로 추정했으나, ground truth
검증 결과 `merge_group_id`/`request_ids_json`/`deleted_item_ids`/`node_id`/`split_only`/
`target_shard_id`는 **이미 1.4.0에 모두 존재**합니다(1.4.0↔HEAD `task_queue.go` 바이트 동일).
따라서 PART A는 task_queue 컬럼을 추가하지 않습니다(추가했어도 멱등 no-op).
